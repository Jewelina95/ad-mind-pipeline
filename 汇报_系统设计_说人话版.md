## 一、问题先讲清楚

我们做 AD（阿尔兹海默症）监测，其实卡在三个地方：

第一，**量表诊断不灵敏**。MMSE 在 MCI 阶段灵敏度只有 18%，MoCA 又特别受教育水平影响。而且这都是"快照式"的——医生看到的是患者那 30 分钟的状态，看不到日常生活里那些细微的变化。

第二，**金标准太贵也太慢**。Amyloid PET 一次 3000-7000 美元，做不起。腰穿创伤性大，老人家受不了。血液 p-tau217 刚获批不久，跨平台标准化还没做好。

第三，**确诊太晚**。从症状出现到确诊平均要 2-5 年。等确诊了，Lecanemab 这些新药的最佳窗口期已经过了。

所以我们想做一个**可穿戴手套 + AI 系统**，从两端同时解决：硬件端连续采集 EDA、PPG、IMU、语音四种信号；AI 端用多智能体把这些信号转成医生能用的临床报告。

## 二、整个系统怎么搭起来的——一图概览

```
┌─────────────────────────────────────────────────────────────┐
│   生成器 v2.1            真实健康 baseline      专家知识库  │

│      │                          │                    │      │
│      └─────────┬────────────────┘                    │      │
│                ▼                                     │      │
│         Analyzer (纯统计)                            │      │
│           ↓ DataFact[]                              │      │
│         KnowledgeRouter ────────────────────────────┤      │
│           ↓ knowledge_text                          │      │
│         3 个 LLM Agent                              │      │
│         (Physio + Behavior + Clinical)              │      │

```

整个流水线**4 层**，左边输入有**3 股流**，最后出**2 份报告**。每一层我都讲一下。

## 三、生成器我们已经做出来了——v2.1

我们的生成器叫 **ad-synthetic-generator v2.1**，公开网址在 `jewelina95.github.io/ad-synthetic-generator/`。

它干一件事：**给 5 个虚拟 AD 患者各生成 30 天的多模态数据**。每个患者有：
- 30 天的传感器数据（IMU/PPG/EDA）
- 30 天 EMA 自评（情绪/睡眠/焦虑/精力，每天 3 次）
- 周量表（MMSE/MoCA/PHQ-9，每周 1 次）
- 月度临床 Note（医生的文字记录）
- BPSD 事件（徘徊/激越/幻觉等，按文献概率注入）

**为什么这个生成器有意义？三个关键设计：**

1. **不是噪声生成器**。我们用 5 个 OpenNeuro 数据集（n=112 真实老年人）做了**分布对标**——用 KS 检验确认我们合成的 MMSE 分布和真实 AD 队列没有统计差异。这个数字是从真实患者算出来的，不是拍脑袋的。

2. **跨模态一致**。整个生成过程由一个 `progression` 变量 [0,1] 驱动——传感器变差的同一天，EMA 情绪也降，量表分也跌，note 也写"主诉记忆问题"。**模拟了真实疾病的全模态同步退化**。

3. **5 种退化模式**。不是所有患者都线性退化的——有的 stepwise（突然恶化），有的 plateau（稳一段又跌），有的 fluctuation（波动），有的 acute_event（突发事件）。这让我们的 demo 比单调的"匀速退化"更接近真实临床。

**生成器的任务到这里就完成了。它不参与判断，只产数据。**

## 四、Analyzer 层——把数据变成"医生能看得懂的事实"

数据有了，下一步要让 AI "看懂"它。这一层叫 Analyzer。

**关键设计：Analyzer 不调 LLM**。

为什么？因为 LLM 不擅长算数。如果你直接把 30 天 × 288 行 × 26 列的 CSV 喂给 Claude，它会算错 z-score、读漏数据、漏掉趋势。所以我们让 numpy 和 scipy 把所有计算干完，Claude 只看"已经算好的事实"。

具体做 5 类检测：

| 类型 | 检测什么 | 公式 |
|---|---|---|
| **outlier** | 单时点异常 | z-score > 2.0 |
| **trend** | 30 天趋势 | linregress p<0.05 且 \|r\|>0.3 |
| **comparison** | 前后对比 | 变化幅度 > 10% |
| **difference** | 双任务代价 | DTC > 20% |
| **extreme** | 跨模态签名 | EDA + HR + IMU 同时异常 = sundowning/agitation |

每条 fact 长这样（举例 P01 患者）：
```
F32 [comparison] [survey] severity=1.00
"MMSE 由 30.0 变化至 11.2 (Δ=-18.8)"
```

**P01 跑下来 Analyzer 出了 34 条 fact**——8 条 outlier、22 条 trend、3 条 comparison、1 条 DTC。这些都是"客观事实"，没有任何主观解读。

## 五、KnowledgeStore 层——文献和专家采访固化的可检索知识库

这一层是我们项目的"专家大脑"。

**包含 35 条 JSON 知识条目**，每条都有：
- 临床特征 / 患病率 / 机制
- 传感器签名 / 阈值
- 干预方案
- **证据等级 (A/B/C/D) + PMID**

知识来自三个地方：
1. **文献综述**：Buracchio 2010、Verghese 2013、Iaboni 2022 这些经典 AD 论文
2. **李医生采访**：26 个临床实操问题的真实回答
3. **临床指南**：GDS 7 阶段、Jack 2024 AT(N) 框架

**关键设计：知识检索由数据驱动，不让 Agent 自由发挥**。

具体怎么做？fact 的 tags 自动决定查什么知识：
- 看到 IMU 模态的 fact → 查传感器知识里 IMU 部分
- 看到 sundowning pattern → 查 BPSD 日落综合征条目
- 看到患者诊断分期 → 查对应 GDS 阶段标准

这样 Agent 永远拿到的是**和当前事实精准匹配的知识**，不会"空引用"或者瞎编。

---

## 六、Agent 层——3 个 LLM Agent，每个负责一摊事

这是真正调 LLM 的地方。三个 Agent，对应 MDT 多学科团队的三个角色：

### PhysioAgent（生理信号专家）
- **角色**：神经内科 + 康复 + 言语治疗师
- **看什么**：IMU 步态 / PPG 心率 / EDA 皮电 / 语音特征
- **输出什么**：biological insight（步态衰退、HRV 下降、自主神经退化）
- **干什么**：不做分期，只描述传感器层面的临床观察

### BehaviorAgent（行为情绪专家）
- **角色**：精神科 + 行为分析师
- **看什么**：EMA 自评 / BPSD 事件 / 活动模式 / EDA 反应性
- **输出什么**：psychological / behavioral insight（情绪低落、激越前驱、睡眠紊乱）
- **特殊任务**：信号消歧——HR 升高到底是激越还是运动？是访客刺激还是日落综合征？

### ClinicalAgent（临床综合专家）
- **角色**：MDT 主持的高年资神经科医生
- **看什么**：上面两位的全部 insight + 量表 + clinical note
- **输出什么**：分期倾向 + 鉴别诊断 + 干预优先级 + 转诊建议
- **关键约束**：不能直接"诊断"，只能"建议倾向"，置信度 < 0.7 时禁止给具体分期

## 七、最关键的一句话——可追溯

整个系统最值得讲的设计，就是**可追溯**。

**每一条 insight 都必须列出它依据的 fact_id 和 knowledge_id**。Agent 不许虚构数字，不许引用未提供的知识。

这意味着医生看到任何一条结论，都可以反查：

```
ins_physio_001 "步速 30 天持续下降, 跌至 0.78 m/s"
└── 来自 fact: F03 (Analyzer trend 检测) + F05 (outlier 检测)
    └── 来源: P01/sensor/walking_normal_dayXX.csv (30 天)
└── 引用知识: SENSOR_IMU_001 + NORM_GAIT_001
    └── 文献: Buracchio 2010 (PMID:20505129, A 级证据)
└── 置信度: 0.88
    └── 因 trend r=-0.91 强 + 知识 A 级 + 模式经典
```

**这是临床落地的硬性要求**。AI 给的结论，医生必须能反查到原始数据和文献依据，否则没法用。

---

## 八、Narrator 层——把 insight 变成医生家属能读的报告

最后一层。Narrator 干三件事：
1. **Threader**：按 alert 等级 + 置信度排序
2. **Insight Card 渲染**：两段式（客观观察 + 加粗的临床含义）
3. **Chart Spec 自动生成**：每条 insight 配一张图的 JSON 配置

输出**两份报告**：
- **医生版**：完整 fact 列表 + 推理链 + 引用 + 干预建议
- **家属版**：红黄绿灯 + 行动建议（"今天建议你做的事"）

## 九、跑通的真实测试结果——P01 患者

理论讲完了，给一个真实的例子。

我们跑了 **P01 患者：68 岁男性，教育 12 年，linear 退化模式，30 天数据**。

**结果**：

| 阶段 | 输出 |
|---|---|
| Generator | ✅ 30 天 sensor + EMA + surveys + 2 BPSD 事件 |
| Analyzer | ✅ **34 条 DataFact** |
| 3 Agents | ✅ **15 条 Insight** (Physio 6 + Behavior 5 + Clinical 综合) |
| Narrator | ✅ 医生版 + 家属版 + 4 张证据卡 |

**最终判断**：🟡 **MCI → mild_AD 倾向（置信度 0.78）**

证据链：
- 步态：步速 30 天 1.05 → 0.78 m/s，接近 Verghese MCR 阈值（0.8）
- 自主神经：HRV 下降 28%
- 情绪：PHQ-9 由 1.7 升至 9.9，EMA 情绪 6.3 → 2.0
- BPSD：Day 24 出现 45 分钟徘徊，Day 28 出现 30 分钟激越

**建议**：
- 🔴 立即启动 GPS 安全监控
- 本周内启动音乐疗法 + 认知训练 + 睡眠卫生
- 转诊做血液 p-tau217 + 头颅 MRI 确认 AD 病理

这套报告**任何一条结论都能反查到原始数据**。

## 十、为什么这样设计——3 个核心原则

1. **LLM 不算数**——所有定量计算给 numpy 干，LLM 只做临床解读。这彻底规避 LLM 算错数字。

2. **知识检索由数据驱动**——不是让 Agent 想查啥就查啥，而是 fact 的 tags 自动决定查什么知识。这保证事实和知识必然配对。

3. **可追溯**——每条结论引用 fact_id + knowledge_id，医生可反查到原始数据 + 文献。这是临床落地的硬性要求。

## 十一、成本和性能

每次评估：
- **总耗时 ~13 秒**（Analyzer 2s + 2 Agent 并发 4s + ClinicalAgent 4s + Narrator 3s）
- **4 次 LLM 调用**，约 6000 input + 2000 output token
- **sonnet-4 ~$0.025 / 次**
- 每周 1 次稳定期评估 → **~$1.3 / 月 / 患者**

每月一千多块人民币 / 患者，临床能负担。
